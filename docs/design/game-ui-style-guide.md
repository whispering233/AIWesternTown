# Game UI Style Guide

## 1. Purpose

这份文档定义 `AIWesternTown` 游戏内主界面的当前 UI 规范基线。它服务于 `apps/web` 的工程实现，但仍属于设计参考层：设计原型与文档先稳定布局和视觉语言，工程代码再按 view model 落地。

当前工程实现参考：

- `design/game-ui-system/design.html`
- `design/game-ui-system/design.png`
- `apps/web`

## 2. Current Direction

### 2.1 Visual Tone

当前方向是浅色纸面、细网格、低饱和西部卷宗感。界面不走营销页或装饰性 dashboard，而是像一张可操作的调查桌面：

- 浅暖背景
- 细边框与分隔线
- serif 品牌与叙事文本
- 小号 mono 状态标签
- 暖色行动强调
- 克制阴影，尽量用留白和边线建立层级

### 2.2 Visual Thesis

主界面应像“西部小镇调查卷宗 + 文本冒险控制台”：玩家聚焦中间叙事，左边回看状态与人物，右边决定移动与空间理解。

### 2.3 Stable Tokens

工程实现应优先稳定这些语义 token：

- `--bg`
- `--surface`
- `--fg`
- `--muted`
- `--border`
- `--accent`
- `--accent-soft`
- `--accent-line`
- `--font-display`
- `--font-body`
- `--font-mono`

## 3. Layout System

### 3.1 Canvas

桌面端主界面使用整屏应用壳：

- 顶栏高度：`64px`
- 外边距：`22-26px`
- 栏间距：`22px`
- 主体高度：占满顶栏以下空间

移动端可降级为纵向堆叠，不强行保留三栏。

### 3.2 Shell Grammar

桌面端采用三栏母版，比例固定为：

`Context Rail : Narrative Rail : Map Rail = 1 : 3 : 1`

三栏职责：

| 区域 | 职责 | 不应承载 |
| --- | --- | --- |
| 左栏 Context | 玩家状态、日志、NPC 人物卡、后续预留 tab | 当前回合主操作、自由输入 |
| 主栏 Narrative | 地点场景标签、生成式叙事、角色对话、选项、自由输入 | 完整地图、完整历史日志、调试面板 |
| 右栏 Map | 总地图、当前位置、移动选项 | LLM trace、系统调试、长篇剧情正文 |

## 4. Top Bar

顶栏只保留品牌与会话状态。

保留：

- 品牌名
- connection 状态
- session 标识

不提供：

- 左栏显示/隐藏按钮
- 右栏显示/隐藏按钮
- 调试入口按钮

顶栏的目标是让玩家知道应用与会话状态，不承担布局控制。

## 5. Main Narrative Rail

主栏从上到下固定为：

1. 地点场景描述标签
2. 场景叙事文本
3. 聊天框
4. 选项块
5. 自由输入块

### 5.1 Scene Tags

场景标签用于回答“我在哪里、当前是什么时间/模式/风险”。标签必须短，不写解释性长文。

### 5.2 Narrative Text

叙事文本由生成式渲染或 view model 摘要提供。工程层只展示，不在 UI 内反推世界真相。

### 5.3 Dialogue Window

聊天框用于 NPC 间对话、玩家与 NPC 对话、旁白式即时后果。完整历史日志应进入左栏“日志”，主栏只展示当前交互附近的信息。

### 5.4 Choice Block

选项块只承载生成式或规则浮出的当前机会动作。当前版本无骰子检定机制，因此选项文案不应出现检定 UI 或成功率 UI。

移动不放在这里，移动统一进入右栏地图的“去往地点”。

### 5.5 Free Input

自由输入是补充入口，放在最底部。它允许玩家表达非模板动作，但仍需要后续通过本地宿主解析为受控命令。

## 6. Left Context Rail

左栏默认使用 tab，当前固定三类：

- `状态`
- `日志`
- `人物`

后续可预留更多 tab，但不要在第一版把所有页面族入口都堆出来。

### 6.1 状态

状态 tab 承载：

- 当前位置
- 世界 tick / 时间
- run mode
- 携带物
- 当前风险或张力

### 6.2 日志

日志 tab 承载短摘要式事件流：

- 玩家输入
- 宿主接收
- 世界后果
- 关键人物反应

单条日志只保留可扫描摘要，长详情进入后续详情页。

### 6.3 人物

人物 tab 承载 NPC 角色卡：

- 名称
- 当前公开身份或玩家视角关系
- 简短状态/线索

角色卡允许折叠，但不应变成完整 Character Sheet。

## 7. Right Map Rail

右栏为地图栏，当前分成上下两块。

### 7.1 上方地图区

上方使用两个 tab：

- `总地图`
- `当前位置`

`总地图` 展示已知地点节点与路线关系。`当前位置` 展示当前场景描述、出口、风险、线索等短信息。

### 7.2 下方移动区

下方固定为“去往地点”，展示可移动地点列表。

移动按钮触发受控移动命令，不在前端直接改变世界真相。

## 8. Interaction Rules

- UI 内按钮只提交命令或切换本地显示状态。
- 前端不复制世界规则，不臆造世界状态。
- 地图节点、选项与自由输入都必须通过 view model 或命令工厂进入宿主链路。
- 调试信息从主界面右栏移除；后续如需要调试页，应作为独立页面或 devtools 入口。

## 9. Component Guidance

工程组件应尽量按职责切小：

- `GameShellLayout`：三栏壳层与顶栏
- `LeftPanelSlot`：左栏 tab 与状态/日志/人物列表
- `MapPanel`：右栏地图 tab 与移动列表
- `SceneFeed`：主栏对话/即时后果窗口
- `PlayableLoopPanel`：主栏选项块
- `CommandComposer`：自由输入块

组件只消费 `ShellViewModel` 或局部回调，不跨层读取 core、host、SQLite 或 provider。

## 10. Iteration Rules

后续 UI 变更先回答：

1. 这块信息属于左栏、主栏还是右栏？
2. 它是否服务当前回合？
3. 它是否需要持久回看或检索？
4. 它是否会把调试信息重新塞回玩家主界面？

如果某块内容不能明确归位，先更新设计文档或原型，不直接塞进工程实现。

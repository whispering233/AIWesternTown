# Game UI Layout Blueprints

## 1. Purpose

这份文档定义 `AIWesternTown` 游戏内 UI 的版式蓝图，重点回答两个问题：

1. 某一类页面应该长成什么骨架
2. 某一类信息应该放进哪一栏

它与 `game-ui-style-guide.md` 的关系是：

- `style guide` 负责视觉语言和结构语法
- `layout blueprints` 负责页面骨架和扩展边界

## 2. Desktop First Shell

桌面端优先采用三栏母版：

- 左栏：`World Rail`
- 主栏：`Play Surface`
- 右栏：`System Rail`

推荐尺寸：

| 区域 | 宽度建议 | 职责 |
| --- | --- | --- |
| World Rail | `320-360` | 世界状态、事件流、日志、索引 |
| Play Surface | `760-840` | 当前回合、主行动、即时反馈 |
| System Rail | `280-320` | 系统摘要、trace、调试、次级上下文 |

统一外层规则：

- 设计基准宽度：`1600`
- 主容器最大宽度：`1520`
- 左右安全边距：`40`
- 栏间距：`24-32`

## 3. Placement Decision Rules

任何一块内容进入页面前，先通过下面的归位规则判断：

| 问题 | 应放位置 |
| --- | --- |
| 这是“当前回合最需要玩家处理的事”吗 | 主栏 |
| 这是“已经发生或持续发生的世界变化”吗 | 左栏 |
| 这是“需要回看、检索、归档”的沉淀信息吗 | 左栏 |
| 这是“系统运行、trace、debug、模式信息”吗 | 右栏 |
| 这是“与当前页相关，但不该打断主行动”的补充信息吗 | 右栏 |

## 4. Main Shell Blueprint

主游玩页始终遵循这条主轴：

`Hero -> Primary Action Frame -> Immediate Consequence -> Focused Input`

具体要求：

- `Hero` 只负责当前场景摘要，不承载长背景
- `Primary Action Frame` 是主栏最强交互区
- `Immediate Consequence` 只保留最近 `1-2` 条强反馈
- `Focused Input` 作为自由输入补充入口，不抢占主位

左栏默认固定三类标签页：

- `状态`
- `事件流`
- `日志`

右栏默认固定两类角色：

- 产品期：系统摘要、运行模式、调试信息
- 后续产品化：目标、风险提示、相关人物、相关地点

## 5. Page Family Matrix

| Page Family | 主任务 | 主栏骨架 | 左栏重点 | 右栏重点 |
| --- | --- | --- | --- | --- |
| Main Shell | 推进当前回合 | Hero / Actions / Consequence / Input | 状态、事件流、日志 | 系统信息、trace |
| Scene Detail | 深读当前场景 | 场景总览 / 场景分区 / 在场角色 / 可探索对象 | 同步保留事件流 | 风险、相邻地点、相关角色 |
| Character Sheet | 深读角色 | 身份卡 / 已知事实 / 关系 / 最近动态 | 日志、相关事件、线索 | 风险标签、关联地点 |
| Consequence View | 阅读关键后果 | 结果摘要 / 影响扩散 / 新线索 / 下一步出口 | 事件流上下文 | 相关目标、系统说明 |
| Journal Timeline | 纵向归档与检索 | 时间线 / 调查条目 / 线索 ledger | 过滤器、索引 | 统计、标签说明 |
| Settlement Overview | 查看据点与区域 | 地图摘要 / 区域分区 / 可去地点 / 关键 tension | 地点日志、传闻 | 区域风险、旅行提示 |
| Investigation Board | 长线调查与推理 | 线索组 / 人物关联 / 未解问题 / 当前假设 | 原始事件流 | 规则说明、当前目标 |

## 6. Family Blueprints

### 6.1 Main Shell

适用场景：

- 默认游玩页
- 玩家正在做当前回合决策

主栏节奏：

1. 场景摘要
2. 当前行动
3. 即时反馈
4. 自由输入

### 6.2 Scene Detail

适用场景：

- 玩家需要停下来读懂一个具体场景
- 当前区域有多个对象、人物或探索点

主栏建议顺序：

1. 场景总览
2. 区域拆分
3. 在场角色
4. 可探索对象
5. 返回行动入口

### 6.3 Character Sheet

适用场景：

- 玩家需要整理角色信息
- 当前角色正在成为调查关键点

主栏建议顺序：

1. 身份卡
2. 已知事实
3. 关系网络
4. 最近动态
5. 相关线索

### 6.4 Consequence View

适用场景：

- 某个事件值得被单独展开
- 当前回合触发了关键后果或阶段变化

主栏建议顺序：

1. 结果摘要
2. 影响扩散
3. 新增线索
4. 下一步入口

### 6.5 Journal Timeline

适用场景：

- 玩家想回看事件链
- 系统需要提供沉淀式档案体验

主栏建议顺序：

1. 时间线主体
2. 调查条目
3. 线索 ledger

左栏不再只是概览，而应升级为：

- 标签过滤
- 角色过滤
- 地点过滤

### 6.6 Settlement Overview

适用场景：

- 世界扩展到多个据点和区域
- 玩家需要决定下一站去哪里

主栏建议顺序：

1. 据点摘要
2. 区域分区
3. 可去地点
4. tension / 风险提示

### 6.7 Investigation Board

适用场景：

- 游戏后期需要支撑复杂线索组织
- 玩家需要把人物、地点、线索联结起来

主栏建议顺序：

1. 当前案件概述
2. 线索分组
3. 人物关系
4. 未解问题
5. 当前假设

## 7. Overflow Rules

为了保护主栏的专注度，以下规则默认生效：

- 主栏首屏只保留一个 `hero`
- 主栏首屏最多出现两个“较重”的信息块
- 如果某块信息需要连续滚动超过两个大分组，优先拆出详情页
- 完整历史事件流不回流主栏
- 可检索和可沉淀的信息优先沉到左栏或单独页面

## 8. Left Rail Interaction Rules

左栏默认是“世界信息容器”，而不是“第二主栏”。

统一规则：

- 默认使用 tabs 切换 `状态 / 事件流 / 日志`
- tabs 之间的信息层级要明确，不混写
- 单条事件只保留短摘要
- 可展开的长详情进入专页，不在 tabs 内无限增长

## 9. System Rail Rules

右栏不是展示所有边角信息的回收站。

统一规则：

- 只放次级但有价值的信息
- 一页最多 `2-3` 个主题块
- 不让右栏承担主行动
- 如果右栏内容成为玩家高频阅读中心，说明它应该上移到主栏或拆成详情页

## 10. Expansion Workflow

以后每增加一个新页面，先回答下面四个问题：

1. 它属于哪一个 `page family`
2. 它是否还在三栏母版里
3. 左栏 tabs 是否沿用，还是需要升级为过滤器
4. 主栏是否仍然保持“单任务”阅读和交互节奏

只有这四个问题都回答清楚之后，才进入视觉细化和工程实现。

# AI Western Town 总设计稿

## 1. 文档定位

本文档是 `AIWesternTown` 项目的母设计稿，用于一次性定义项目的产品目标、世界框架、核心玩法、NPC 认知系统和技术边界。它不是宣传稿，也不是详细实现说明书，而是后续所有实现设计和内容设计的上位规范。

本文档的主要读者是：

- 项目发起人
- 负责实现的代理或工程师

本文档中的结论分为三类：

- `已定设计`：当前阶段已经锁定的方向，后续实现默认以此为准
- `实现建议`：为了降低风险和提高可执行性而给出的默认做法
- `待讨论问题`：必须在后续子文档中进一步收敛的内容

当前状态补充：

- NPC 认知框架第一版已基本完成
- 世界推进与状态仿真第一版已基本完成

## 2. 项目定位

### 2.1 一句话定义

`已定设计`

这是一款以“西部世界式主题乐园”为灵感来源的单人文字冒险叙事沙盒游戏。玩家作为外来者进入一座拟真西部小镇，通过观察、试探、介入和挑拨，影响一群由大模型驱动思考和行动的 NPC，使每一局都演化出不同的人际关系、秘密揭露和结局。

### 2.2 核心卖点

`已定设计`

- NPC 不是固定台词机，而是持续运行的“角色主体”
- 玩家不是沿着单一剧情树前进，而是在扰动一个会自行演化的社会系统
- 每局差异的来源不是随机文本，而是 NPC 记忆、关系和计划的动态变化

### 2.3 第一版目标

`已定设计`

第一版的核心目标不是做完整商业化游戏，而是验证一套可稳定运行的 NPC 循环思考与世界演化框架，证明以下三件事成立：

1. NPC 可以在后台持续思考并相互影响
2. 玩家介入能够显著改变局势，而不是只触发预设分支
3. 同一初始设定在不同局中可以导向不同过程和不同结局

### 2.4 第一版非目标

`已定设计`

第一版暂不追求：

- 复杂画面表现
- 3D 场景探索
- 多人模式
- 完整商业化内容体量
- 配音、多模态或大规模开放世界

## 3. 玩家体验目标

### 3.1 玩家身份

`已定设计`

玩家在第一版中扮演“新来者/游客”。这个身份的好处是：

- 可以自然接触不同 NPC，而不需要先解释玩家为何拥有强权限
- 适合观察和扰动小镇的社会关系
- 既能保持探索感，也方便后续加入隐藏身份、阵营选择或角色背景

### 3.2 单局体验

`已定设计`

单局游戏目标时长为 30-90 分钟。一次标准游玩体验应当包含以下阶段：

1. 进入小镇并认识当前局的基本氛围
2. 接触数名关键 NPC，发现他们表面身份和潜在矛盾
3. 通过谈话、试探、挑拨、协助或隐瞒改变局势
4. 目睹 NPC 关系和事件链逐步升级
5. 进入某个收束结局，并获得局后总结

### 3.3 为什么每局都不同

`已定设计`

每局差异主要来自以下因素叠加：

- 初始随机种子带来的角色状态和事件时机差异
- NPC 对同一事件的不同解读和记忆写入
- 玩家介入顺序和方式不同
- NPC 在局中产生的关系变化、误会、联盟和敌对
- 同一秘密在不同时间被不同人得知，导向不同链式反应

## 4. 世界框架

### 4.1 世界类型

`已定设计`

世界采用“主题乐园式西部小镇”的设定框架。它看起来像一座经典西部边境小镇，但其本质是一个人为设计、可重复运行、拥有角色脚本与隐秘层的叙事场域。

这意味着世界同时具备两层特征：

- 表层：酒馆、旅店、治安所、马厩、教堂、街道、火车站等西部叙事符号
- 深层：角色秘密、隐藏动机、循环日常被扰动后的异常行为、局中逐渐显露的系统性裂缝

### 4.2 地点结构

`实现建议`

第一版不采用真实地图或自由移动，而是采用“文本场景节点”组织世界。每个地点是一个具备以下属性的场景节点：

- 地点名
- 公开描述
- 常见出入角色
- 可能发生的事件类型
- 线索密度
- 是否适合作为冲突升级点

建议首批核心地点控制在 6-8 个，足以支撑社交和事件流转。

### 4.3 角色规模

`已定设计`

第一版同时活跃的核心 NPC 控制在 6-10 人，推荐默认值为 8 人。这样可以形成最基本的社会网络，又不至于让调度、调试和成本失控。

### 4.4 角色类型

`实现建议`

第一版的角色池应覆盖不同社会位置和冲突源，例如：

- 小镇管理者
- 治安角色
- 服务业角色
- 边缘人物
- 外来者
- 隐藏破局者

角色之间必须天然带有至少一种关系张力，例如：

- 旧怨
- 利益冲突
- 隐藏合作
- 身份隐瞒
- 忠诚摇摆

## 5. 核心玩法循环

### 5.1 玩家循环

`已定设计`

玩家的主循环是：

1. 观察当前场景和最近世界变化
2. 选择接触对象或行动方向
3. 通过选项或自由输入表达意图
4. 触发 NPC 反应和世界推进
5. 读取后果并重新评估局势

### 5.2 输入模式

`已定设计`

玩家输入采用混合模式：

- 常规状态下以选项为主，保证可读性和节奏稳定
- 关键时刻允许自由文本，提升开放性和扮演感

`实现建议`

自由文本不能直接进入世界规则层，必须先经过意图解析，映射为受控动作，例如：

- 提问
- 指控
- 劝说
- 转移话题
- 观察细节
- 前往地点
- 打断他人
- 隐瞒信息

### 5.3 玩家影响方式

`已定设计`

玩家不直接操纵 NPC 或世界状态，而是通过下列方式影响系统：

- 改变 NPC 对自己或他人的印象
- 把信息从一个角色传递给另一个角色
- 激化或缓和两个角色间的冲突
- 暴露、掩盖或误导与秘密有关的线索
- 在关键时刻站队，从而推动结局倾向

### 5.4 单局收束

`已定设计`

第一版不是无限沙盒，而是单局叙事沙盒。每一局都必须有可判定的结局收束条件。建议结局触发来源包括：

- 关键秘密被揭露
- 主冲突升级到不可逆阶段
- 主要角色关系网络发生决定性断裂或联盟形成
- 玩家触发强干预结局
- 时间耗尽，小镇以某种状态稳定或失控收束

局后必须生成：

- 本局事件摘要
- 关键角色命运变化
- 玩家造成的主要影响

## 6. NPC 核心设计

### 6.1 设计原则

`已定设计`

NPC 不是对话容器，而是小镇中持续运行的主体。每个 NPC 都应具备：

- 身份与公开人设
- 长期目标
- 当前处境
- 记忆能力
- 关系网络
- 行动能力
- 反思与调整能力

### 6.2 认知循环

`已定设计`

NPC 的标准认知循环定义为：

`Perceive -> Appraise -> Update Working Memory -> Goal Arbitration -> Action Selection -> Act -> Reflect -> Compress`

含义如下：

- `Perceive`：感知当前场景、他人行为、玩家输入和近期事件
- `Appraise`：对感知结果做主观评价，判断威胁、机会与社交意义
- `Update Working Memory`：把当前最重要的问题、风险和机会维持为短时关注焦点
- `Goal Arbitration`：在多个短期与长期目标之间选出当前主导目标
- `Action Selection`：把主导目标落到当前最合适的一步动作
- `Act`：执行结构化动作并生成权威结果
- `Reflect`：在特定时机回顾近期事件，形成新的判断
- `Compress`：压缩记忆，保留关键事实与关系变化

当前阶段补充约定：

- 上述 8 个阶段已经在 `doc/30-npc-cognition-framework.md` 中完成第一版正式展开
- 认知主链路已经形成 `感知 -> 评价 -> 焦点更新 -> 目标仲裁 -> 动作选择 -> 执行 -> 反思 -> 压缩` 的闭环
- 长期记忆读取、流程图、数据库结构和内部接口已分别拆到 `doc/35`、`doc/36`、`doc/37`、`doc/38`
- 后续在母稿中不再重复展开阶段级细节，具体实现默认以对应子文档为准

### 6.3 NPC 状态层

`已定设计`

每个 NPC 当前至少维护四层核心状态：

1. 身份层
   - 姓名
   - 角色定位
   - 公开形象
   - 秘密
   - 长期目标
   - 红线或禁忌
   - 核心驱动力

2. 工作记忆层
   - 当前焦点与 concern
   - 当前主导问题、风险和机会
   - 当前意图附近的短时状态
   - 最近一轮认知链路沉淀出的关注项

3. 长期记忆层
   - 情景记忆
   - 关系经验
   - 线索与传闻
   - 对玩家的持续印象
   - 可供后续检索的 retrieval summary

4. 社交信念层
   - 对其他 actor 的信任、怀疑、恐惧、依赖和利用判断
   - 当前局部关系变化
   - 后续 `Perceive / Appraise / Reflect` 可读取的社会判断切片

补充说明：

- `Working Memory` 是认知主链路中的唯一标准短时状态结构
- tick 内的读取缓存与检索账本属于运行时上下文，不单独视为持久状态层
- 长期记忆与社交信念在母稿层面已经从笼统“记忆层”拆开，以避免实现时混淆读取边界

### 6.4 行动原则

`已定设计`

NPC 不直接输出“任意自然语言行为”，而是先产出结构化动作，再由叙事层转化为玩家可见文本。这样可以避免模型绕开规则系统。

第一版的动作链路进一步约定为：

- `Action Selection` 只负责选出当前最合适的一步结构化动作
- `Act` 负责生成权威执行结果、状态改动和事件记录
- 如果 NPC 决定暂不显性动作，应通过“带意图的待机 + hold 执行模式”表达，而不是跳过动作选择
- 对玩家可见的台词、姿态和叙事文本只能建立在已确认的执行结果之上

第一版建议的动作类别包括：

- 前往某场景
- 对某人发起对话
- 回避某人
- 调查某条线索
- 隐瞒或销毁信息
- 透露或试探信息
- 威胁、挑拨、拉拢、协助
- 带意图的待机

### 6.5 涌现标准

`已定设计`

第一版判断 NPC “像活的”不是看文笔，而是看是否满足以下标准：

- 同一事件会被不同 NPC 以不同角度记住
- NPC 关系会因局中事件真实改变
- NPC 会围绕自己的立场和记忆持续做出后续行为
- 玩家不干预时，NPC 之间仍会自行演化出冲突或合作

## 7. 世界推进与仿真规则

### 7.1 时间机制

`已定设计`

世界采用“离散 tick + 玩家视角实时化”的推进方式。系统内部以固定步长推进，而玩家感受到的是世界持续运行。

`实现建议`

首版默认：

- `1 次玩家有效动作 = 1 次局部世界步进 = 1 个 worldTick`
- 玩家阅读文本、查看日志和纯 UI 操作不推进世界
- 对话期间世界不暂停，但后台只以低速继续
- 每个局部 tick 仍从相关 NPC 的 `Cycle Prefetch` 起步，再进入各自被授权的认知阶段
- 世界采用“玩家动作显式推进时间，玩家视角感受为持续运行”的折中方案

### 7.2 事件驱动

`已定设计`

世界核心变化通过事件链组织，而不是全量重算；调度器则通过“场景泡泡式分层调度”决定谁在本 tick 内获得完整行动机会。每个关键行为都必须沉淀为可追踪的世界事件，例如：

- 某人得知了某事
- 某人公开指控另一人
- 某角色被激怒
- 某秘密传播到新对象
- 某角色转而信任或怀疑玩家

补充约定：

- 玩家动作必须先结算，再允许 NPC 响应
- 世界按 `foreground / near_field / far_field` 三层分度推进
- 第一版默认每 tick 最多只让少数前台 NPC 跑完整链路，近场 NPC 只做有限补步，远场 NPC 主要进入 backlog
- 调度运行模式固定为 `free_explore / focused_dialogue / interrupted / settle`
- `Act` 阶段必须把关键结果沉淀为结构化事件记录和状态改动
- `Reflect` 和 `Compress` 不直接改写世界事实，只消费执行结果并生成后续认知影响
- 世界推进闭环默认采用 `Act -> World Event -> Reflect -> Compress -> 下一轮检索/决策`
- 强打断只允许在当前轮动作结算完成后抢占下一次玩家输入，不允许打断已提交动作的半程结算
- 同一 NPC 在同一个 `worldTick` 内不得重复获得完整链路资格

### 7.3 真实状态与表现文本分离

`已定设计`

系统中必须区分：

- 世界事实
- 面向玩家的叙事表达

同一事件可以有文学化描述，但其底层状态变化必须是结构化且可追踪的。这样做是为了保证：

- 调试可行
- 回放可行
- 后续内容扩展可控

### 7.4 存档和复盘

`实现建议`

首版会话应保留以下可序列化信息：

- 当前世界状态
- NPC 身份、目标、工作记忆、社交信念和长期记忆快照
- 事件日志
- 调度器运行模式、场景泡泡分层和优先队列状态
- 对话线程、强打断状态和远场 backlog
- 认知阶段输出摘要和关键调试元数据
- 长期记忆读取索引摘要
- 随机种子
- 模型输出摘要

这样后续才能支持：

- 结局总结
- 异常排查
- 复盘和回放

## 8. 大模型职责边界

### 8.1 设计原则

`已定设计`

第一版不采用“全靠模型即时生成一切”的方案，而是采用“规则层 + LLM 层”的混合架构。

### 8.2 规则层负责

`已定设计`

规则层负责以下内容：

- 世界推进调度
- tick 驱动、阶段编排和长期记忆读取权限控制
- 场景泡泡分层、运行模式切换和调度优先级排序
- 每 tick 预算与上限控制
- 强打断硬判、对话挂起与恢复判定
- 动作合法性判断
- 会话生命周期管理
- 场景切换与基础状态更新
- 工作记忆、社交信念、长期记忆等权威状态写入
- 感知候选构造、基础打分、候选过滤和预算控制
- 胜负和结局判定
- 调试、存档和日志

### 8.3 LLM 层负责

`已定设计`

LLM 层负责以下内容：

- 角色主观解释和高语义复杂度判断
- 对规则层候选结果做有限语义精修
- 在受控结构下生成对话文本、简短表现文本和摘要
- 对局势提示、强打断提示和远场摘要做受限文本渲染
- 辅助判断记忆重要性、反思结论和压缩摘要

明确边界：

- LLM 不直接拥有世界状态写入权
- LLM 不得绕过动作结构直接创造世界事实
- LLM 渲染文本不能反向定义规则层真相

### 8.4 Provider 抽象

`已定设计`

模型接入必须从第一版开始就抽象成 provider 层，支持：

- 云端模型
- 本地模型
- mock 测试模型

这不是为了第一版就兼容一切，而是为了避免后续系统被某一家模型接口锁死。

### 8.5 成本控制

`实现建议`

持续运行的 NPC 仿真如果不做预算控制，很快会失控。第一版默认加入：

- 每个 NPC 的重推理频率限制
- 每局推理预算限制
- 超预算时的降级路径

降级方式可以包括：

- 复用近期摘要
- 跳过低优先级反思
- 让部分 NPC 进入轻量行为模式

## 9. 技术方向摘要

### 9.1 系统形态

`已定设计`

第一版默认采用浏览器前端 + 后端权威世界模拟的架构。

前端负责：

- 文本展示
- 玩家输入
- 当前局势摘要
- 局后总结呈现

后端负责：

- 世界状态
- NPC 调度
- 世界调度器运行状态
- 事件总线
- 对话线程与强打断状态
- 模型调用
- 日志和存档

### 9.2 状态权威

`已定设计`

世界真相只存在于后端。前端不直接拥有规则真相，只展示“当前对玩家开放的信息”和“叙事化表达”。

### 9.3 调试需求

`已定设计`

第一版必须从一开始就保留调试视图能力，至少能查看：

- 当前世界时间
- 当前调度运行模式
- 活跃 NPC 的当前意图
- 最近关键事件
- 当前 `foreground / near_field / far_field` 分层结果
- 本 tick 的调度资格分配、抑制原因和预算命中情况
- 强打断候选、已选中断原因以及对话线程状态
- 某 NPC 最近一轮各认知阶段的结构化输入输出
- 长期记忆预取命中、阶段追加检索结果和读取账本
- 某 NPC 最近一轮感知、计划、执行和反思结果
- 远场 backlog 和近场补步队列

没有这类调试能力，后续将无法判断 NPC 是“真的在推理”还是“只是看起来在说话”。

## 10. 内容生产原则

### 10.1 角色设计原则

`实现建议`

第一版角色设计不要追求数量，而要追求每个角色都具备以下最小信息闭环：

- 公开身份
- 私密动机
- 可被揭露的秘密
- 至少两条关系张力
- 对玩家的潜在利用方式

### 10.2 冲突设计原则

`已定设计`

冲突不能只靠“坏人作恶”来驱动，而应当来自以下更适合涌现的来源：

- 误解
- 利益冲突
- 身份隐瞒
- 忠诚冲突
- 自我保护
- 对同一线索的不同解释

### 10.3 文本表现原则

`实现建议`

第一版文本表现应服务于“局势理解”和“角色可信度”，而不是只追求华丽文风。文案的优先级应当是：

1. 状态清楚
2. 意图可信
3. 角色有区分度
4. 文学风格统一

## 11. 第一版成功标准

`已定设计`

当出现以下情况时，可以认为第一版方向成立：

1. 玩家能明显感受到 NPC 会记住事情，并据此改变态度和行动
2. 玩家在不同时间介入同一冲突，会得到不同后果
3. 玩家即使不强干预，也能观察到 NPC 自行推动事件发展
4. 同一套初始设定重复开局后，过程和结局存在明显差异
5. 开发者能通过日志和调试视图解释大多数关键行为来源

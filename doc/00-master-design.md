# AI Western Town 总设计稿

## 1. 文档定位

本文档是 `AIWesternTown` 项目的母设计稿，用于定义第一版的产品目标、体验方向、系统边界和文档导航。它只保留总纲级结论，不重复子文档中的实现细节。

本文档的结论分为三类：

- `已定设计`
- `实现建议`
- `待讨论问题`

当前状态：

- 世界与剧情线设计第一版已完成
- 核心玩法循环设计第一版已完成
- 场景分区与可见性设计第一版已完成
- NPC 认知框架第一版已完成主链路设计
- 世界推进与状态仿真第一版已完成
- 物品系统与数据模型设计第一版已完成
- LLM 集成与 Prompt Builder 第一版已完成

## 2. 项目定位

### 2.1 一句话定义

`已定设计`

这是一款以“西部世界式主题乐园”为灵感来源的单人文字叙事沙盒游戏。玩家作为外来者进入一座拟真西部小镇，通过观察、试探、介入和挑拨，影响一群持续运行的 NPC，使每一局演化出不同的人际关系、秘密揭露和结局。

### 2.2 第一版目标

`已定设计`

第一版优先验证以下三件事：

1. NPC 可以持续思考并相互影响。
2. 玩家介入能够显著改变局势。
3. 同一初始设定可以导向不同过程和不同结局。

### 2.3 第一版非目标

`已定设计`

第一版暂不追求：

- 复杂画面表现
- 3D 场景探索
- 多人模式
- 大规模开放世界
- 商业化内容体量
- 配音与多模态演出

## 3. 玩家体验目标

### 3.1 玩家身份与单局体验

`已定设计`

玩家在第一版中扮演“新来者/游客”。单局目标时长为 `30-90` 分钟，标准体验应包含：

1. 进入小镇并感知当前氛围。
2. 接触关键 NPC，识别表面身份与潜在矛盾。
3. 通过观察、侦查、谈话和站队改变局势。
4. 目睹关系和事件链逐步升级。
5. 进入某个可解释、可复盘的结局。

### 3.2 每局差异来源

`已定设计`

每局差异主要来自：

- 初始随机种子
- NPC 对事件的不同解读与记忆写入
- 玩家介入顺序与方式
- 关系变化、秘密传播和联盟/敌对演化

## 4. 世界与内容框架

### 4.1 世界结构

`已定设计`

世界采用“主题乐园式西部小镇”的设定框架。第一版以文本场景节点组织世界，首批核心地点建议控制在 `6-8` 个，核心 NPC 建议控制在 `6-10` 人。

补充约定：

- 世界的主导航单位仍是场景节点
- 关键高价值场景允许在节点内部继续拆成少量 `scene partition`
- 场景分区的职责是稳定观察、旁听、被注意到和同场景站位差异，不把第一版扩展成几何级地图玩法

### 4.2 双层内容系统

`已定设计`

第一版采用“双层内容系统 + 单一世界运行内核”：

- `基础角色涌现层`
  - 由地点、角色卡、关系种子、秘密和基础事件模板组成
  - 即使不加载剧情线，也必须能自然生成冲突和收束
- `可选剧情线层`
  - 由 `1` 条主线骨架和若干支线组成
  - 只提供叙事偏置，不直接命令 NPC

### 4.3 会话形态

`已定设计`

第一版支持两种内容装配形态：

- `Freeform Sandbox`
  - 只加载基础角色涌现层
- `Narrative Sandbox`
  - 在前者之上额外挂载主线与支线

两者共享同一套世界运行内核、事件结算链路和收束框架。

### 4.4 物品与容器层

`已定设计`

第一版中的物品不是纯叙事装饰，而是世界中的一等实体。规则层维护唯一的物品占有真相，玩家与 NPC 都通过统一动作语义围绕物品产生拿取、转移、藏匿、使用和发现后果。

补充约定：

- 物品、容器和藏匿位都属于世界真相的一部分
- 玩家与 NPC 共享同一套基础物品动作语义
- 观察、记忆、事件和调度层只消费结构化物品事实，不以文本反推真相

## 5. 核心玩法循环

### 5.1 玩家主循环

`已定设计`

玩家主循环为：

`进入地点 -> 粗观察 -> 主动侦查 -> 浮出机会 -> 原子动作或社交卷入 -> 读取后果 -> 重新评估`

### 5.2 输入与影响方式

`已定设计`

第一版采用“选项为主、自由文本为辅”的混合输入模式。自由文本必须先经过意图解析，再映射为受控动作。

玩家不直接操纵世界状态，而是通过以下方式影响系统：

- 改变 NPC 印象
- 传播或遮掩信息
- 激化或缓和冲突
- 站队、试探、协助或误导

### 5.3 单局收束

`已定设计`

第一版不是无限沙盒。单局必须能够因秘密揭露、冲突升级、关系断裂、时间耗尽或强干预结局而收束，并生成事件摘要、角色命运变化和玩家影响总结。

## 6. NPC 核心设计

### 6.1 设计原则

`已定设计`

NPC 不是对话容器，而是持续运行的主体。每个 NPC 至少具备：

- 身份与公开人设
- 长期目标与核心驱动力
- 记忆能力
- 社交关系与局部信念
- 行动能力
- 反思与调整能力

### 6.2 认知主链路

`已定设计`

NPC 的标准认知循环定义为：

`Perceive -> Appraise -> Update Working Memory -> Goal Arbitration -> Action Selection -> Act -> Reflect -> Compress`

当前约定：

- 8 个阶段已在子文档中完成第一版展开
- 工作记忆是唯一标准短时状态结构
- 长期记忆、社交信念和认知日志都使用结构化结果驱动

### 6.3 状态层

`已定设计`

每个 NPC 当前至少维护四层核心状态：

- 身份层
- 工作记忆层
- 长期记忆层
- 社交信念层

### 6.4 行动原则

`已定设计`

NPC 先产出结构化动作，再由叙事层转化为玩家可见文本。`hold` 是执行模式，不是独立动作类型。

### 6.5 特殊长动作

`已定设计`

睡眠与顿悟作为 NPC 内部长动作存在。它们不冻结世界，只在正常结算时触发一次深度处理；若中途被打断，则整次深处理任务丢弃。

## 7. 世界推进与仿真规则

### 7.1 时间推进

`已定设计`

第一版采用“离散 `worldTick` + 玩家视角实时化”的推进方式。默认规则为：

- 玩家命令会先进入调度器；其中只有 `consumesTick = true` 的有效动作才推进 `worldTick`
- 阅读文本和纯 UI 操作不推进时间
- 对话期间世界不暂停，但后台低速继续

### 7.2 分层调度

`已定设计`

世界采用“场景泡泡式分层调度”：

- `foreground`
- `near_field`
- `far_field`

调度器负责决定谁在当前 tick 获得完整链路、谁只做轻量补步、谁进入 backlog。

场景内的可见性与旁听约束由 `scene partition topology` 提供更细一层的空间真相，但不改变 `foreground / near_field / far_field` 的世界级调度框架。

### 7.3 事件与强打断

`已定设计`

世界核心变化通过结构化事件组织，而不是全量重算。玩家动作先结算，再允许 NPC 响应。

强打断规则：

- 只在当前轮动作结算完成后生效
- 只抢占下一次玩家输入
- 不打断已提交动作的半程结算

### 7.4 调度边界

`已定设计`

调度器负责：

- 运行模式切换
- 场景泡泡分层
- 预算和上限控制
- 强打断判定
- 对话挂起与恢复

调度器不直接写入 NPC 认知事实、长期记忆正文或世界真相。

## 8. LLM 职责边界

### 8.1 总体原则

`已定设计`

第一版采用“规则主导，LLM 附着”的保守方案。规则层先算资格、候选、合法性和状态写入，LLM 只在少数高语义复杂节点按需参与。

### 8.2 规则层负责

`已定设计`

规则层负责：

- 世界推进与调度
- 动作合法性与状态写入
- 事件与结局判定
- 预算控制与降级决策
- 调试、日志和存档

### 8.3 LLM 层负责

`已定设计`

LLM 负责：

- 主观解释与有限语义精修
- 对话、局势提示和可见文本渲染
- 摘要、反思和压缩辅助

明确边界：

- LLM 不直接拥有世界状态写入权
- LLM 不得直接决定强打断、动作资格和结局
- LLM 文本不能反向定义规则层真相

### 8.4 Provider 与 Prompt Builder

`已定设计`

模型接入必须通过 provider 抽象。所有 LLM 调用都先生成 `PromptSpec`，再经 role-aware compiler 编译成最终消息，由 parser 收回结构化结果。

## 9. 技术与调试原则

### 9.1 技术方向

`已定设计`

第一版默认采用浏览器前端 + 后端权威世界模拟的架构。

- 前端负责文本展示、玩家输入和局后总结
- 后端负责世界状态、NPC 调度、模型调用、日志和存档

### 9.2 调试要求

`已定设计`

第一版必须保留调试视图，至少能查看：

- 当前 `worldTick` 与运行模式
- 最近关键事件
- 当前分层调度结果
- 强打断与对话线程状态
- 某 NPC 最近一轮认知阶段输入输出
- 长期记忆读取命中与账本

没有这类调试能力，后续无法稳定解释 NPC 行为来源。

## 10. 内容生产原则

### 10.1 总体原则

`已定设计`

第一版内容系统至少应显式支持以下对象：

- `CharacterCard`
- `RelationshipSeed`
- `ItemTemplate`
- `ContainerTemplate`
- `NarrativeLine`
- `EventTemplate`
- `EndingTemplate`

这些对象的职责是支撑可配置装配，而不是把具体内容硬编码进流程。

### 10.2 设计取向

`实现建议`

内容设计应优先保证：

1. 状态清楚
2. 动机可信
3. 关系有张力
4. 文本有区分度

## 11. 第一版成功标准

`已定设计`

当满足以下条件时，可以认为第一版方向成立：

1. 玩家能明显感受到 NPC 会记住事情，并据此改变态度和行动。
2. 玩家在不同时间介入同一冲突，会得到不同后果。
3. 玩家即使不强干预，也能观察到 NPC 自行推动事件发展。
4. 同一套初始设定重复开局后，过程和结局存在明显差异。
5. 开发者能通过日志和调试视图解释大多数关键行为来源。
